# Life-Rhythm 项目文档

> 本文档记录了项目从构想到方案确定的完整思考过程，供后续对话快速恢复上下文。

---

## 一、起点：从 life-tracker 出发

已有项目 `life-tracker`（位于 `/Users/lpy/Project/life-tracker`）是一个每日回顾的 Web 应用，使用飞书多维表格作为数据后端。

新的想法是：在每日回顾之外，还需要一个**今天要做什么、做了没有**的执行追踪系统。两者定位互补：

- **life-tracker**：每天结束，回顾今天发生了什么（向后看）
- **life-rhythm**：每天开始/进行中，执行计划好的事情（向今天看）

---

## 二、需求梳理过程

### 2.1 初始想法（用户原话整理）

用户描述了三个大类的每日固定安排：

**第一大类：属灵生活（Basic-Life）**
- 灵修：固定阅读内容（如马太福音第一章，内容固定不变）+ 目标项（分享给他人、阅读相关材料、完成应用，每项打勾，后续统计）
- 读经、查经、背诵、交流

**第二大类：生活相关**
- 身体：健身、室内运动、户外运动
  - 有周期性目标：每天室内运动15分钟、每周健身2次、每周户外运动1次
  - 目标会随时间调整：下个月户外运动2次、健身3次
- 技能培养：英语（每天阅读15分钟、音标等）、项目应用经验

**第三大类：提升类**
- 不紧急但重要的进阶内容（学习新语言等）
- 习惯类：每日运动、喝水、作息统计

**另有大型计划：**
- 一年读一遍圣经（具体到每天、每月、每季度的进度统计）
- Vibe coding 项目计划

### 2.2 需求分析后的核心发现

所有内容本质上只有三种数据类型：

| 类型 | 定义 | 例子 |
|------|------|------|
| **习惯 (Habit)** | 有周期性目标的可重复行为 | 每天室内运动15分钟、每周健身2次 |
| **清单项 (Checklist)** | 某习惯下的细分检查项，逐日勾选 | 灵修下的分享/阅读材料/完成应用 |
| **计划 (Plan)** | 有总量目标的长期任务 | 一年读一遍圣经（1189章） |

区分这三种类型是整个数据库设计的基础。

## 三、数据库选型的思考过程

### 3.1 飞书多维表格的问题

讨论后确认飞书多维表格不适合本项目，原因：
1. **数据主权风险**：平台依赖，公司有倒闭风险，数据不在自己手中
2. **性能问题**：多维表格是协作工具而非数据库，API 调用开销大
3. **查询能力不足**：周期汇总、累计进度、完成率等复杂统计几乎无法实现

### 3.2 三种自建方案对比

| 方案 | 费用 | 优点 | 缺点 |
|------|------|------|------|
| 本机搭建 | 免费 | 完全掌控 | 关机无法访问、硬件故障风险 |
| 购买 VPS | 约50-80元/年 | 全天在线、数据自控 | 需要自己运维 |
| Supabase（托管） | 免费 | 零运维、性能好、数据可导出 | 数据在第三方服务器 |

### 3.3 最终决定

**选择 Supabase 免费套餐。**

理由：数据全是文本，免费额度（500MB）远够用；查询性能是纯数据库级别；数据随时可完整导出，不存在锁定风险；零运维成本。

### 3.4 飞书数据迁移

暂缓。等 life-rhythm 稳定运行后，再考虑将 life-tracker 的飞书数据迁移到 Supabase，届时两个项目完全统一在同一数据库。

## 四、与 life-tracker 合并的方案

### 4.1 可行性

完全可行。两个模块在同一个 Next.js 应用内只是不同路由，数据来源不同但互不干扰：
- `/review` → life-tracker 模块（飞书 API）
- `/plan` 或 `/` → life-rhythm 模块（Supabase）

共用导航、UI 组件、样式系统，用户体验连贯。

### 4.2 开发策略

**先在 `/Users/lpy/Project/life-rhythm` 单独开发**，验证功能完整后再合并进 life-tracker。好处是不污染现有项目，life-tracker 继续正常使用。

## 五、最终数据结构设计

### 5.1 分类层级

```
属灵生活 (L1)
  └── 灵修 (L2)
  └── 读经 (L2)
  └── 查经 (L2)
  └── 背诵 (L2)
  └── 交流 (L2)

生活相关 (L1)
  └── 身体 (L2)
        └── 健身 (L3)
        └── 室内运动 (L3)
        └── 户外运动 (L3)
  └── 技能培养 (L2)
        └── 英语阅读 (L3)
        └── 音标 (L3)
        └── 项目应用经验 (L3)

提升类 (L1)
  └── 语言学习 (L2)
  └── 每日运动 (L2)
  └── 喝水统计 (L2)
  └── 作息记录 (L2)
```

### 5.2 数据库表结构

**分类表 `categories`**
- 自引用父子关系，支持无限层级
- 字段：id、parent_id、name、icon、color、type（spiritual/life/improvement）、sort_order

**习惯定义表 `habits`**
- 字段：id、category_id、name、track_type（boolean/duration/count）、unit（min/次/杯/章）、active

**习惯目标表 `habit_goals`**
- 每条记录对应一个时间段内的目标（支持按月调整，历史记录保留）
- 字段：id、habit_id、target_value、period_type（daily/weekly/monthly）、start_date、end_date

**打卡记录表 `habit_logs`**
- 字段：id、habit_id、log_date、value（实际完成值）、notes

**灵修清单模板表 `spiritual_checklist_items`**
- 灵修下的固定子项定义
- 字段：id、habit_id、name、is_fixed_plan（是否固定阅读）、fixed_content（马太福音第一章）、sort_order

**灵修清单完成记录表 `spiritual_checklist_logs`**
- 每天每项的勾选状态
- 字段：id、item_id、log_date、completed、notes

**计划定义表 `plans`**
- 字段：id、category_id、name、plan_type（reading/project/annual）、total_units、unit、start_date、end_date

**计划进度记录表 `plan_logs`**
- 字段：id、plan_id、log_date、units_done、note

**作息记录表 `sleep_logs`**
- 字段：id、log_date、sleep_at（入睡时间戳）、wake_at（起床时间戳）、quality（1-5分）、notes

### 5.3 关键设计决策

1. **目标随时间变化**：新增记录而非修改原有记录，历史统计不受影响
2. **灵修清单专用设计**：先做专用，有其他习惯出现类似需求再抽象通用
3. **作息单独一张表**：需要存两个时间点（入睡/起床），与习惯表结构不同
4. **周起始日**：全局设置，影响"每周N次"类目标的统计

## 六、统计功能一览

| 统计项 | 说明 |
|--------|------|
| 今日完成率 | 今天计划的习惯完成了几项/几成 |
| 周目标达成 | "每周N次"类目标本周达成情况 |
| 连续打卡天数 | 某习惯坚持了多少天未中断 |
| 灵修子项月统计 | 本月分享几次、完成应用几次 |
| 计划累计进度 | 圣经读到哪，按日/月/季/年汇总 |
| 习惯目标对比 | 实际完成 vs 当期目标 |
| 作息规律 | 平均睡眠时长、起床时间分布 |

## 十、项目名称说明

- **life-tracker**：每日回顾，看昨天
- **life-rhythm**：每日执行，看今天——了解今天要做什么，还有哪些没做，然后去行动

名称"rhythm"取生活节奏之意，契合习惯的周期性和日复一日的执行感。
